2 Main Types of UI

Where you are fitting into an existing interface

  • Many tasks are handled for you (e.g. menus, widgets, ...)
  • Your UI should look familiar or 'native'. If you don't users will be confused and will not have a good experience
  • You generally will be distributing your platform widely. You must therefore think about things like localization (l10n), internaltionalization (i18n) and accessibility (a11y)

Where you are providing everything yourself

  • You'll need to handle drawing everying and responding to raw input data. There may be code libraries to help with this
  • You will generally have a much freer hand to design your interface the way you want, and choosing what you elements you want to use
  • Often you will be targeting a very specific set of users, and you know exactly how they will interact with the UI, so you don't need to cater to all different user needs.

Both have their challenges

Text

  • Text is hard
  • Easy: Ow now brown cow
  • Easyish: mais Mémé aimait même, mes mets...
  • Hard: यह देवनागरी है, और यह हिब्रू में नमस्ते है: שלום מימין לשמאל
  • (My text editor completely failed on the last one, but browsers tend to have really good support for alternate scripts)

Text encoding

  • Massive subject, but the upshot is everyone uses Unicode and UTF-8 (except Microsoft of course, who use a slightly wrong UTF-16)
  • The thing to remember is "Unicode is the way of recording all text from any languages, and UTF-8 is how to actually store it on disk efficiently"

Text shaping

This is turning a string of text into a sequence of glyphs and locations to draw them

  • In English, there is almost always a 1-1 correspondence between characters and glyphs
  • Not so in other languages
  • Arabic (العربية) normally has 4 ways to draw each character, for start, middle, end of a word, and on its own
  • Oh yeah, Hebrew and Arabic are written backwards. How does selecting text work when you have languages in both directions? People have actually solved this.
  • kerning, emoji 😃, ...

Text drawing

Glyphs must be turned from a list of curves into actual pixels at a given size to be drawn.
  • Antialiasing (including subpixel)
  • Hinting
  • 4k screens mean we may be able to consign some techniques to the dustbin of history

Input handling

  • Traditionally, the main sources of input were mouse and keyboard. Now we have many devices, including speech, touchscreen, accessible devices, and more.
  • Most input methods can be represented as either a stream of characters, or a pointer to parts of the screen. This greatly simplifies the job of application developers: they do not have to think about the great complexity here; it can be handled by UI libraries
  • Again, scripts come to bite us again here: CJK languages (Chinese, Japanese and Korean) have far too many characters to have a keyboard key for each one
  • They usually use Imput Method Editor (IME) software. All I will say here is that you should support it

Accessibility

I've run out of time, but this should be its own talk anyway. There are many different disabilities, and many different technologies to enable full access to computing for people with any of them. A very quick list: color blindness and gradient selection, blindness, screen readers and tree-based interface representation, motor problems and keyboard-only navigation (tab through input elements).

Accessibility is something that you cannot ignore as an application developer, the same way you can ignore other stuff I've talked about. High-contrast mode, easily understandable text, and much besides is in the domain of the application developer.
